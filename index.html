Betterbox.es

This site is intended to build on and complement the work of Gareth Rushgrove. Gareth started Vagrantbox.es, a large and growing list of publicly-available Vagrant base boxes. This is great thing!

However...

There are some useful conventions I would like to propose base box builders and maintainers to follow. Betterbox.es is a curated list to encourage the adoption of these conventions.

In the tables below, you will find boxes conforming to a new versioned format, based on and extending the pattern set by the base boxes Opscode provides as part of Bento.

Base Base Boxes

These are intended to be minimal installations of the OS. Beyond that they should only include what is necessary to meet the requirements outlined in the Vagrant Docs.

Fields

Box
The name for a box follows the same basic structure as the opscode bento boxes (organization namespace, OS name and version) with the addition of a version number for the box itself. This version number has two components: the first represents the version of the veewee definitions used, the second is incremented whenever a new box is built (with updated packages and guest additions).
OS
Self-explanatory.
Arch
Self-explanatory.
Maintainer
Whether an individual or an orgnization, each box should have a designated maintainer.
Definitions
Link to the definitions used (e.g. a tag in a git repo).
Build Date
Self-explanatory
Changelog
Link to the heading in the changelog corresponding to this version of the box. The changelog can include changes to definitions used (major version releases), and changes to packages and guest additions (for minor version releases).

Derived Base Boxes

These are base boxes with commonly used packages and configuration pre-provisioned for faster build time.

If you use a default configuration that is time-consuming to provision, it might make sense to export it as a base box. Base boxes like this that are generic enough that they might be useful to others can be added to this list.

Each derived base box should explicitly list its parent (a versioned box from above). Ideally it should also include a link to the configuration management source used (Puppet manifests, Chef recipes, etc.).
